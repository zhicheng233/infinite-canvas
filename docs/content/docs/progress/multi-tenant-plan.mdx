# 多租户积分商用改造大纲

## 项目现状

当前版本 v0.4.0：纯前端个人工具，所有数据存浏览器本地，API Key 也存浏览器直接请求 OpenAI 兼容接口。无认证、无多租户、无后端。

v0.0.8 曾经有过完整的用户账户+积分体系，在 v0.4.0 被整体移除。前端仍残留 `credits.tsx` 等积分 UI 基础设施。

## 一、后端重建（Go + Gin + GORM）

### 1.1 多租户模型层
- `Tenant` 表（id, name, domain, plan, status, created_at）
- `User` 表（id, tenant_id, username, password_hash, role, status）
- `CreditAccount` 表（id, tenant_id, user_id, balance, total_earned, total_spent）
- `CreditTransaction` 表（id, account_id, type, amount, balance_after, ref_type, ref_id, note）
- `CreditPricing` 表（id, tenant_id, model, credits_per_unit, unit_type）

### 1.2 认证鉴权层
- JWT 登录/注册（用户名+密码）
- OAuth 可选
- 租户隔离中间件（每个请求校验 tenant 归属）
- 角色权限（super_admin / tenant_admin / user）

### 1.3 积分管理层
- 积分充值接口
- 积分扣减（生成前预扣、成功确认、失败退回）
- 积分流水查询
- 管理员积分定价配置

### 1.4 AI API 代理层
- 前端不再直连 AI API，改为经过后端代理
- 后端校验积分余额 → 扣减 → 代理请求 → 返回结果
- 多模型渠道配置（由租户管理员在后端配置）
- 请求日志（计费审计）

### 1.5 画布数据服务层
- 画布项目 CRUD
- 图片/媒体文件上传到后端存储
- WebDAV 同步保留为可选

## 二、前端改造

### 2.1 认证与租户选择
- 登录/注册页面
- Token 管理与自动刷新
- 全局 useUserStore 接入后端用户信息

### 2.2 积分系统 UI 接入
- 复活 credits.tsx，接入后端积分余额 API
- 画布顶部显示积分余额
- 生成按钮展示预估算力消耗
- 积分不足提示与充值入口
- 积分流水页面

### 2.3 API 层重构
- 新增认证拦截器（自动附带 JWT）
- auth.ts / credits.ts / canvas-sync.ts
- image.ts / video.ts / audio.ts 改为走后端代理

### 2.4 数据持久化迁移
- 新用户直接使用后端存储
- 现有本地数据提供"导入到云端"功能
- 保留 localforage 作为离线兜底

### 2.5 管理后台
- 租户管理、用户管理、积分定价配置
- 积分充值、用量统计

## 三、部署与运维

- Docker Compose 一键部署（前端 + 后端 + MySQL + Redis）
- Nginx 反向代理
- 请求限流、API Key 不暴露给前端

## 四、执行顺序

| 阶段 | 内容 | 优先级 |
|------|------|--------|
| Phase 1 | 后端骨架 + 用户认证 + 多租户模型 | P0 |
| Phase 2 | 积分系统 + AI API 代理 | P0 |
| Phase 3 | 前端认证 + 积分 UI 接入 | P1 |
| Phase 4 | 画布数据云端存储 + 图片上传 | P1 |
| Phase 5 | 管理后台 | P2 |
| Phase 6 | 数据迁移工具 + 离线兜底 | P2 |
| Phase 7 | 支付对接 | P3 |
