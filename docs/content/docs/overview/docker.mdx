# Docker 部署

当前版本采用 `Next.js + Go backend + MySQL + Redis` 的组合部署，适合单机或单台云服务器运行。

## 服务结构

- `app`：前端站点，对外提供 `3001`
- `backend`：Go 后端，对外提供 `18080`
- `mysql`：业务数据库，仅容器内访问
- `redis`：缓存与验证码等辅助能力

## 首次部署

```bash
git clone https://github.com/GeQainZz/infinite-canvas.git
cd infinite-canvas
chmod +x scripts/init-env.sh
./scripts/init-env.sh
```

也可以手动按需补齐 `.env`：

```env
MYSQL_ROOT_PASSWORD=replace-this-password
MYSQL_DATABASE=infinite_canvas
JWT_KEY=replace-this-jwt-secret
API_KEY_ENCRYPTION_KEY=replace-this-aes-secret
REGISTRATION_CREDITS=0
NEXT_PUBLIC_API_URL=https://your-domain/backend-api
NEXT_PUBLIC_DOC_URL=https://docs.canvas.best
```

启动：

```bash
docker compose up -d --build
```

说明：

- `app` 服务会通过根目录 `Dockerfile` 自动构建前端镜像 `infinite-canvas-app:latest`
- `backend` 服务会通过 `backend/Dockerfile` 自动构建后端镜像
- 首次部署与后续更新都建议直接使用 `docker compose up -d --build`

## 常见部署报错

### `docker compose up -d` 提示 `Artifact Registry API has not been used`

这类报错大多数不是 Google Artifact Registry 真的需要启用，而是 Docker 没有走本地构建，转而把 `infinite-canvas-app:latest` 当成远程镜像去拉取。

按下面顺序执行即可：

```bash
docker compose up -d --build
```

如果想先确认镜像是否存在，可以执行：

```bash
docker images | grep infinite-canvas-app
```

没有输出就说明前端镜像还没构建。

## 首次初始化

### 1. 自动创建初始管理员

当数据库里还没有任何用户，且 `.env` 已配置以下变量时，后端首次启动会自动创建一个 `super_admin`：

```env
INIT_ADMIN_USERNAME=admin
INIT_ADMIN_PASSWORD=replace-this-admin-password
INIT_ADMIN_DISPLAY_NAME=系统管理员
```

这个初始化只会在“空数据库首次启动”时执行一次；后续重启不会重复改动已有用户。

### 2. 登录后台完成系统配置

登录后进入 `/admin/api-config`，至少完成以下配置：

- 上游 `API Base URL`
- 上游 `API Key`
- 模型目录
- 模型状态
- 积分计费
- 视频模型路由

### 3. 如果你跳过了自动创建

如果首次启动时没有配置 `INIT_ADMIN_*`，系统仍可启动，但不会自动生成管理员。  
这种情况下，仍然可以先注册普通账号，再手动把该用户提升为 `super_admin`。

## 访问地址

- 前端：`http://服务器IP:3001`
- 后端 API：`http://服务器IP:18080/api`

如果你已经有正式域名，推荐改为：

- 前端：`https://your-domain/`
- 后端 API：`https://your-domain/backend-api`

## 更新发布

推荐更新命令：

```bash
git pull
docker compose up -d --build
```

如果只更新后端：

```bash
docker compose up -d --build backend
```

如果只更新前端：

```bash
docker compose up -d --build app
```

如果修改了 `.env` 里的 `NEXT_PUBLIC_API_URL`：

```bash
docker compose up -d --build app
```

## 域名反向代理推荐

推荐正式环境把前后端统一收口到同一个 HTTPS 域名，并把前后端接口前缀彻底区分开：

- 前端页面与 Next.js 内置接口：`https://your-domain/`、`https://your-domain/api/*`
- Go 后端业务接口：`https://your-domain/backend-api/*`

对应做法：

1. `.env` 中设置：

```env
NEXT_PUBLIC_API_URL=/backend-api
```

2. 重建前端：

```bash
docker compose up -d --build app
```

3. 反向代理：

- `/` 转发到 `127.0.0.1:3001`
- `/api/` 转发到 `127.0.0.1:3001`
- `/backend-api/` 转发到 `127.0.0.1:18080`

仓库已提供示例：

- Nginx：`deploy/nginx/hmgai.cc.conf`
- Caddy：`deploy/Caddyfile.example`

4. 当前 `docker-compose.yml` 已把后端端口收口到 `127.0.0.1:18080:18080`，默认只允许本机反向代理访问。
5. 推荐把 `.env` 中的 `NEXT_PUBLIC_API_URL` 直接写成 `/backend-api`，这样浏览器侧和代理层职责更清晰。

### Caddy 部署示例

如果你服务器已经在使用 Caddy，推荐直接这样配置：

1. 修改项目 `.env`：

```env
NEXT_PUBLIC_API_URL=/backend-api
```

2. 重建前后端：

```bash
docker compose up -d --build app backend
```

3. 把 `deploy/Caddyfile.example` 内容写入你的 Caddy 配置，例如：

```caddy
hmgai.cc {
    encode gzip zstd
    request_body {
        max_size 100MB
    }

    @frontendApi path /api/*
    handle @frontendApi {
        reverse_proxy 127.0.0.1:3001
    }

    @backendApi path /backend-api/*
    handle @backendApi {
        reverse_proxy 127.0.0.1:18080
    }

    handle {
        reverse_proxy 127.0.0.1:3001
    }
}
```

4. 重载 Caddy：

```bash
caddy fmt --overwrite /etc/caddy/Caddyfile
caddy reload --config /etc/caddy/Caddyfile
```

5. 验证：

- 打开 `https://hmgai.cc/`
- 在浏览器开发者工具中确认登录请求走的是 `https://your-domain/backend-api/auth/login`
- 访问 `https://your-domain/api/prompts?page=1&pageSize=20` 应返回提示词列表，而不是 404

说明：

- `/api/*` 由 Next.js 处理，当前主要用于提示词聚合等前端内置接口
- `/backend-api/*` 才是 Go 后端业务接口，包含鉴权、画布、积分、代理等能力

如果你的 Caddy 不是 systemd 管理，也可以用：

```bash
caddy validate --config /etc/caddy/Caddyfile
```

## 迁移到另一台服务器

如果你要把当前线上环境迁移到另一台新服务器，建议按下面顺序：

### 1. 复制代码与 `.env`

把当前项目代码和线上 `.env` 一并带过去。

### 2. 启动空环境

```bash
docker compose up -d --build
```

### 3. 导出旧库数据

在旧服务器执行：

```bash
docker exec ic-mysql mysqldump -uroot -p"${MYSQL_ROOT_PASSWORD}" infinite_canvas > infinite_canvas.sql
```

### 4. 导入新库数据

把 `infinite_canvas.sql` 传到新服务器后执行：

```bash
cat infinite_canvas.sql | docker exec -i ic-mysql mysql -uroot -p"${MYSQL_ROOT_PASSWORD}" infinite_canvas
```

### 5. 重启后端与前端

```bash
docker compose up -d --build backend app
```

### 6. 登录后台复核

重点检查：

- `/admin/api-config` 上游地址和 API Key 是否正确
- 视频模型是否已经配置到正确接口路由
- 模型计费是否完整
- 普通用户是否看不到管理后台

## 当前版本说明

- 用户注册/登录、个人中心、积分体系、画布保存、图片/视频/音频生成记录已走后端存储。
- 用户侧 AI 调用统一走后端代理；普通用户不再自行配置 API Key。
- 管理员在 `/admin/api-config` 统一配置上游 API 和模型计费。
- 只有已配置计费的模型才会出现在前台可选模型中。
- 视频模型支持按模型显式指定接口路由，不同上游可分别走 `/v1/videos`、`/v1/videos` JSON / yijia、`/v1/videos/generations`、`/v1/video/generations` 或 Seedance 路由。

## 生产建议

- 不要把 MySQL 暴露到公网。
- `JWT_KEY` 与 `API_KEY_ENCRYPTION_KEY` 请使用高强度随机字符串。
- 建议给 `3001` 与 `18080` 前面加 Nginx / Caddy，统一做 HTTPS 与反向代理；正式环境优先使用同域名 `/api` 方案。
- 备份至少包含 `mysql_data` 卷和你自己的 `.env`。
- `NEXT_PUBLIC_API_URL` 属于前端构建参数，修改后必须重建 `app` 容器才能生效。
